# Project Policies

プロジェクト全体に常時適用される判断ルール集。タスク中に方針疑問が生じたら本ファイルを参照する。

関連:
- `../CLAUDE.md` — エントリポイント (本ファイルへのポインタを持つ)
- `workflow.md` — 開発フロー（superpowers 計画・branch/PR）
- `product.md` — プロダクト概要・ドメイン
- `../.kiro/specs/<spec>/design.md` — 完了済み spec の設計判断（アーカイブ）

---

## 修正案を提示する際

複数案を出すときは「**症状の隠蔽** / **症状の緩和** / **根本解決**」のラベルを明示する。workaround を採用するなら、なぜ根本解決が取れないのか理由を明示する。ユーザーは根本解決を強く好む。

## リポジトリ構造はその都度の最適形を追求する

「既存ファイル編集を優先・新規追加は最小限」は採用しない。責務分割・凝集度・可読性のために必要なら新規ファイル・新規ディレクトリ作成を躊躇しない。逆に「既存ファイル肥大化の延命」「責務不明確な追加」は避け、本質的な変更を選ぶ。

- 抽象化は「現時点で必要か」で判断する。将来予測で抽象化しない

## 品質ゲートは省略しない

全ゲートをクリアしないと commit しない。

- 「スコープ外なので linter / 型 / テストは飛ばす」は採用しない。必要なら最小修正で通す
- 使い捨てスモークスクリプトで済ませず、最初から pytest テストを書く
- 詳細・各ツールの ignore 方針は `development.md`

## ドキュメントの育て方

- **CLAUDE.md はエントリポイント** — 詳細は本ファイルや `.kiro/`、`docs/` に分離。エントリ自体は短く保つ
- 各タスク完了ごとに「CLAUDE.md / docs/ に追記すべき知見はあるか」をユーザーに確認する
- 既存ドキュメントで参照可能な情報は重複させず、ポインタで繋ぐ
- 古くなった記述は更新・削除する

## Architecture 原則（常時適用）

- Signal データは immutable（読み込み後の変更禁止）。入力ファイルへの書き込みは厳禁（データ改ざん防止）。
- 同期・Formula 等の変換処理は元データを変更せず新しいオブジェクトを生成する。
- Protocol（`typing.Protocol`）でモジュール間インターフェースを定義し疎結合を維持。
- Strategy パターンでフォーマット別パーサーを差し替え可能にする。
- コアロジック（データ処理）と GUI 層を分離し、コアは GUI に依存しない。
- 将来の AD（完全自動運転）スコープ拡張を見据えた拡張性。

## 言語標準

- **文章（docs・設計 spec・コミットメッセージ・PR）**: 日本語。
- **コード識別子（変数・関数・クラス名）・技術用語**: 英語。
- **コードコメント**: 日本語可（WHY を書く）。
- 旧来の spec 生成ルールにあった「英語で書く」規定は廃止（実態と逆だった）。
