# Kiro + Claude Code 併用開発ガイド

本プロジェクトでは **Kiro** (spec-driven 設計 + steering) と **Claude Code** (大規模実装) を併用する。本ドキュメントはその運用ルールと情報フローを明文化したもの。

関連:
- `../CLAUDE.md` — Claude Code のエントリポイント
- `../.kiro/steering/` — Kiro が自動読込するプロジェクトルール
- `../.kiro/specs/` — 機能ごとの要件・設計・タスク
- `policies.md` — プロジェクト方針

---

## ツールの役割分担

| ツール | モード | 得意領域 | 使い分け |
|--------|--------|----------|----------|
| Kiro | Spec | 要件定義・設計・タスク分解 | 新機能の計画段階。requirements → design → tasks の流れ |
| Kiro | Vibe | 探索的コーディング・Q&A・小修正 | 調査・プロトタイプ・単発の質問 |
| Claude Code | — | 大規模実装・リファクタ・CI修正 | tasks.md に沿った連続実装、複数ファイル横断の変更 |

### 判断基準

- **「何を作るか」を決める** → Kiro Spec
- **「どう作るか」の探索** → Kiro Vibe
- **「決まったものを作る」** → Claude Code (tasks.md 消化)
- **バグ修正・CI 修正** → Claude Code (CLAUDE.md 経由で文脈把握)

### 情報配置の原則

| 原則 | 説明 |
|------|------|
| **Single Source of Truth** | 情報は一箇所に書き、他からはポインタで参照する |
| **CLAUDE.md は薄く保つ** | エントリポイントとして最小限のポインタ + ハマりどころのみ |
| **steering は不変ルール** | プロジェクト全体に常時適用される原則。頻繁に変わらない |
| **specs は機能単位** | 機能ごとに独立。完了後もアーカイブとして残す |
| **docs は運用知識** | spec に収まらない横断的な知見・手順 |

## 新機能開発フロー

```
1. Kiro Spec で計画
   └── requirements.md → design.md → tasks.md を生成

2. CLAUDE.md を更新
   └── Phase 表に新 spec を追記 (状況: 進行中、一次情報源: spec パス)

3. Claude Code で実装
   └── tasks.md を番号順 / 依存グラフ順に消化
   └── 各タスク完了時: 品質ゲート実行 → チェックボックス [x] 更新

4. 知見の蓄積
   └── タスク完了ごとに「CLAUDE.md / docs/ に追記すべき知見はあるか」を確認
   └── 肥大化したら docs/ に分離、ポインタを残す

5. Phase 完了
   └── CLAUDE.md の Phase 表を「完成」に更新
```

## CLAUDE.md の設計思想

Claude Code は会話開始時に `CLAUDE.md` を読む。そのため:

- **発見可能性を最優先** — 「どこを見ればいいか」が即座にわかる優先順位表を冒頭に置く
- **重複禁止** — `.kiro/` や `docs/` で確認可能な情報は書かない
- **コードから自明な事実は書かない** — ファイル名・関数シグネチャ等
- **ハマりどころは書く** — 環境依存の罠、暗黙の前提条件
- **品質ゲートコマンドは書く** — 毎回参照するため、ポインタではなく実体を置く

### CLAUDE.md の標準セクション構成

```markdown
# CLAUDE.md
## 情報の探し方 (優先順位)     ← 必須。ポインタ表
## プロジェクト概要             ← 1-2行
## 開発ワークフロー             ← dual-agent-workflow.md へのポインタ
## Phase 状況                  ← 進捗管理
## プロジェクト方針 (要約)      ← docs/policies.md へのポインタ
## 主要コマンド (品質ゲート)    ← 実体 (毎回使うため)
## 開発環境の落とし穴           ← 環境固有のハマりどころ
## ファイル更新ルール           ← コメント方針・spec 更新ルール
```

## .kiro/steering/ の設計思想

Kiro は `.kiro/steering/` 配下の `.md` ファイルを自動的にコンテキストに含める。そのため:

- **プロジェクト全体に常時適用されるルール** を置く
- **頻繁に変わらない情報** に限定する (変更が多いものは specs/ か docs/)
- **基本構成 (4ファイル) + 必要に応じて追加**:

| ファイル | 内容 | 更新頻度 |
|----------|------|----------|
| `product.md` | プロダクト概要・ドメイン知識・設計原則 | 低 (方針変更時のみ) |
| `tech.md` | 技術スタック・コマンド・設定 | 低〜中 (ツール追加時) |
| `structure.md` | ディレクトリ構造・命名規則・モジュール境界 | 中 (構造変更時) |
| `spec-authoring.md` | spec (requirements/design/tasks) 生成時のルール | 低 (方針変更時のみ) |

ルールが特定タイミング (spec 生成時等) でしか効かない場合でも、Kiro 自動読込により常時参照される利点があるため steering に置く価値がある。

## コーディングルール (共通)

以下は言語・フレームワークに依存しない共通ルール。具体的なツール設定は `development.md` を参照。

### 品質ゲート

- コミット前に **全ゲートを通す** (テスト・lint・format・型チェック)
- 「スコープ外だから飛ばす」は採用しない
- 使い捨てスモークスクリプトではなく、最初から正式なテストを書く

### コード品質

- コメントは **WHY** を書く。WHAT は書かない
- 純粋ロジックと I/O を分離する (テスタビリティ確保)
- 抽象化は「現時点で必要か」で判断する。将来予測で抽象化しない
- 責務分割のために新規ファイル作成を躊躇しない

### 修正案の提示

- 複数案を出すときは「**症状の隠蔽** / **症状の緩和** / **根本解決**」を明示
- 根本解決を優先。workaround を採用するなら理由を明示

### ドキュメント

- 情報は一箇所に書き、ポインタで繋ぐ
- 古くなった記述は更新・削除する
- タスク完了ごとにドキュメント追記の要否を確認する
