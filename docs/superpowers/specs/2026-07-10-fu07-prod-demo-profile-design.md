# 設計 spec: FU-07 — 本番想定デモ mf4（`prod` プロファイル）

**日付**: 2026-07-10
**対象**: catalog `SS-FOLLOWUP` の **FU-07**（デモ mf4 をより本番想定に）
**種別**: 開発ツール（`scripts/` のみ・製品コード不変更）
**一次情報源**: [docs/audit-findings-catalog.md](../../audit-findings-catalog.md) の FU-07

## 位置づけ

全6改善サブスペック完了後の実使用フォローアップ FU-01〜09 のうち、**他フォローアップの再現・検証を可能にする土台**として着手する FU-07。本番（ADAS ECU の HILS 評価・XCP フル計測）相当の**大規模チャンネル数**の mf4 を生成し、以下を実機で再現・検証できるようにする:

- **FU-01**（1024列超の展開ダイアログ `ExpansionDialog` がスクロール不能で範囲外にアクセス不能）
- **FU-03**（大容量ロード済みのドック表示切替がフリーズ気味・遅い）

既存の `scripts/generate_demo_mf4.py`（2026-07-04・[spec](2026-07-04-hils-demo-mf4-generator-design.md)）へ**新プロファイル `prod` を追加**する増分。既存 `hils`/`quick`/`smoke` は据え置き。

### FU-08 についての正直な限界（重要）

FU-08（Y軸オートフィットの O(サンプル数) 走査が重い）は**高サンプル数**の信号で顕在化する。`prod` は計測長 120秒・最高レート 10ms のため 1信号あたり最大 **12,000サンプル**にとどまり、オートフィットの O(N) 走査を強くは再現しない。**FU-08 の再現は既存 `hils` プロファイル（1ms×3600s＝約360万サンプル/信号）が適役**。`prod` は FU-01/FU-03 を主眼とし、本 spec では FU-08 を主対象に含めない（catalog の FU-08 は hils で検証）。

## 目標（原 FU-07 の3数値・厳守）

| 指標 | 目標 | 備考 |
|---|---|---|
| ファイル内タイムスタンプ（計測長） | **~120秒** | ユーザー確定（ロード時間ではなく格納タイムスタンプ） |
| 展開後チャンネル数 | **~32万**（LD-14 多次元フラット展開後のスカラー channel 数） | ±20% |
| ファイルサイズ | **~1.5GB** | ±20% |

いずれも実装時に**実測して ±20% 以内**に収める（レート/本数/dtype で微調整）。

### 制約解決（ユーザー決定 A）

「1.5GB／32万／120秒／10ms 50%」は同時成立しない（10ms 半分だけで 16万ch×12,000サンプル×1B ≈ 1.92GB > 1.5GB）。ユーザー決定 **A**: 3数値（1.5GB/32万/120秒）を優先し、**10ms 比率を ~30% に下げ、残りを 100〜500ms に寄せる**（本番 XCP フル計測に近い＝内部変数の大半は低レート・高速ラスタは一部）。

## 非目標

- 製品コード（loader/GUI/VM）の変更。FU-01/03 の**修正**は各フォローアップの別着手（本 spec は再現データの提供のみ）。
- `hils`/`quick`/`smoke` プロファイルの変更。
- FU-08 の再現（上記の限界により hils が適役）。
- 実 DBC/ARXML の取り込み（信号定義・値はスクリプト内生成、既存方針を踏襲）。
- 生成物のコミット（`demo_data/` は gitignore 済み）。

## 生成データ設計

### プロファイル定義

`PROFILES` に `prod` を追加（既存 `Profile(duration_s, chunk_s)` を拡張、下記「実装ノート」参照）:

- `duration_s = 120.0`
- `chunk_s = 5.0`（1チャンクの numpy 生成量を抑えピークメモリを数百MBに保つ）
- 大規模チャンネル構成（`hils` の ~200ch と別系統の group 定義）。

### 32万変数の内訳（混在・アレイ主体）

生の mf4 は**多次元アレイ主体**で、展開後に 32万へ到達させる（生 append 本数を数千に抑え、asammdf のメタデータ・生成時間を現実的にする）。

| 部分 | 概算（展開後） | レート / dtype | 狙い |
|---|---|---|---|
| シナリオ核 | ~200 実信号 | 10ms / float64 | 既存 ACC シナリオ（120秒＝300秒サイクルの部分）を表示。実プロット/カーソル/範囲統計の意味的確認 |
| バルク・アレイ（≤1024列・10ms） | ~96k（~96アレイ×~1000列） | 10ms / uint8 | 物標/状態行列を模した多次元アレイ。**サイズの主ドライバ**（~1.15GB）。多チャンネルで **FU-03** |
| バルク・アレイ（≤1024列・低レート） | ~224k（~224アレイ×~1000列） | 100〜500ms / uint8・int16 | 計数の主ドライバ（~0.3GB）。低レートで軽量・**FU-03** |
| 広幅アレイ（>1024列） | 数十本（例 ~60本×~1500列） | 10ms・100ms / uint8 | LD-14 の 1024列超ガードを発火。**十分な本数（数十〜）で FU-01**（ダイアログのチェックボックスが画面高を超えスクロール不能）。上記バルクの一部を >1024列に置換して総数を保つ |
| バルク・スカラー | ~数千 | 500ms / float64 | XCP 内部変数風の平坦スカラー（現実味の補強・低レートで軽量） |

**サイズ試算**（uint8 アレイ主体）: 10ms 96k×12,000×1B ≈ 1.15GB ＋ 100ms 224k×1,200×1B ≈ 0.27GB ＋ 核/スカラー数十MB ＝ **~1.45GB**（目標 1.5GB ±20% 内）。実装時に実測して本数/列/レートで微調整。

### レート分布

- **10ms: ~30%**（サイズ主因）。
- 100ms / 200ms: 中間（計数の主体）。
- **500ms: 最低レート**（軽量な計数埋め）。

### dtype 戦略

- アレイ（バルク・広幅）: **uint8 主・一部 int16**。量子化した物標/状態値＝本番の格納表現と整合し、かつサイズを抑える（1.5GB を 120秒×32万で満たす鍵）。現行 `_pack_array_channel`（asammdf 8.8.11 で ndim==2 を保つ byte-array 化）を踏襲し、loader が LD-14 で要素展開できる形にする。
- シナリオ核・バルクスカラー: float64（意味的な信号）。

### シナリオ（値の意味づけ）

シナリオ核は既存の ACC 追従シナリオ関数（`veh_spd`/`ttc`/物標軌跡等）を**そのまま再利用**し、120秒ぶん（cruise→lead_decel→cutin の部分サイクル）を表示。バルク・アレイ/スカラーは既存 `ctrl_internal` 系の決定的合成（位相シフト）を流用し、意味は持たないが再現可能。

## 生成の実現性

- **生 append は数千本**（アレイ ~350–400本＋核 ~200＋スカラー数千）。32万本の逐次 append を回避。asammdf のメタデータも生本数ぶんで軽い。
- **チャンク extend**（既存機構）で 5秒チャンクずつ追記。ピークメモリ数百MB。
- **生成時間は分オーダー**の見込み（実測して `docs/development.md` に記載）。
- 乱数はグループ間で独立ストリーム（`np.random.default_rng(seed + group_id)`）、シナリオ値は時刻の決定的関数（チャンク境界で再現性）。既存方針を踏襲。

## 実装ノート（既存コードとの接続）

- `Profile` dataclass は現状 `(duration_s, chunk_s)` のみ。`prod` の大規模構成は**別の group 定義**（`_build_prod_groups()` 等）を要するため、プロファイルごとに「どの group 群を使うか」を分岐できるようにする（例: `Profile` に group ビルダーの識別子を持たせる or プロファイル名で分岐）。既存 `GROUPS` 定数（hils/quick/smoke 共通）と別に prod 用 group を構築。**この構造変更はユーザー承認済み方針の範囲**（`scripts/` 内・製品コード不変更）。
- 広幅アレイ（>1024列）の生成: 既存 `_radar_obj_matrix` 等の (N,K) パターンを一般化した**列数パラメータ付きのアレイ生成関数**を新設。
- バルク・アレイ/スカラーの本数はループ生成（`_ctrl_internal_signals` の拡張）。

## テスト戦略

すべて Layer A / スクリプト単体で実質を尽くせる（実 1.5GB を CI で作らない）。

1. **サイズ/本数の試算ユニットテスト（純関数・実生成なし）**: プロファイル定義（レート×duration×dtype×本数×列）から**期待ファイルサイズ・展開後チャンネル数を算出する純関数**を新設し、`prod` が目標レンジ（1.5GB±20%・~32万±20%）内であることを assert。プロファイル定義が目標から乖離したら RED（数値のドリフト検知）。
2. **CI 極小版（構造検証・秒級）**: `prod` の縮小版（数十アレイ×小列×短時間、`smoke` 同様に tmp 生成）を `Session.load` し、(i) 複数レートの group が存在、(ii) >1024列アレイが1本以上存在、(iii) 展開後チャンネル数が縮小版の期待値、を assert。※実サイズ（1.5GB）は CI 非対象。
3. **シナリオ値の代表 assert**（既存テスト流儀）: 縮小版でも定常区間 `VehSpd ≈ 80` 等、シナリオ核の妥当性を確認。
4. **実機確認手順**（`docs/development.md` に追記・`/run` `/verify` 観測）: `uv run python scripts/generate_demo_mf4.py --profile prod` → `uv run valisync` でロード → **FU-01**（広幅アレイの展開ダイアログがチェックボックス多数でスクロール不能）・**FU-03**（多チャンネルのロード後、ドック表示切替がフリーズ気味）を観測。生成時間・ロード時間・ファイルサイズ・展開後チャンネル数を記録。
