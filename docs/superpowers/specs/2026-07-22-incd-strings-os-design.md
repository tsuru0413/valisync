## 2. アーキテクチャ

### 2.1 `gui/strings.py` — GUI 文言の単一の真実

`src/valisync/gui/strings.py` を新設。**pure Python・Qt 非依存**（`theme/tokens.py` と同じ隔離方針 — headless で全テスト可能）。

- **形式**: モジュールレベル `Final[str]` 定数。エリア別プレフィックスで命名（`MENU_FILE = "ファイル(&F)"`・`DOCK_FILE_BROWSER`・`DIALOG_EXPORT_TITLE`・`STATUS_LOADED = "{source} を読み込みました"` 等）。dict キー参照ではなく属性参照 — typo が mypy/ImportError で loud-fail し、実行時 KeyError の芽がない。
- **テンプレート**: 動的文言は名前付きプレースホルダの `str.format` テンプレート定数。書式付き数値（`{offset:+.3f}`）はテンプレートに書式を内蔵し、散在フォーマッタを引き寄せる（R-06 のオフセット単一化）。
- **重複定義の解消**: 現状の三重/二重定義（File Browser×3面＋参照文・Diagnostics×3面＋参照文3〔main_window 434/436・channel_browser_view:50〕・テーマラベル×2・scope radio・区切りコンボ 2 ファイル）は全て同一定数の参照に置換。ドック名は windowTitle／折りたたみタイトルバー／レールタブが**同じ定数**から合成され、参照句（「「診断」ドックを参照」等）も単一定数から合成する。
- **ニーモニクス込みで一意**: 共有定数の文言はニーモニクス込みで同一文字列（3面共有の文言にニーモニクスは付与しない — §2.4 の適用面規則と一体）。
- **2面共有（メニューバー×コンテキスト/ツールバー）の規則**: 定数は常に**素形**（& なし）を真実とし、メニューバー掲載面のみ `mn(text, key)` ヘルパ（`f"{text}(&{key})"` 合成・strings.py 内の pure 関数）で付与形を合成する。例: 「補間方式」は Analyze メニューでは `mn(S.INTERP_METHOD, "I")`＝「補間方式(&I)」・空白右クリックでは素形。ニーモニクス割当文字は §3 G-46 の表が唯一の出典。
- **VM/workers も strings.py を参照**: viewmodels・workers は Qt 非依存のまま `strings` を import できる（strings.py が pure であることの直接の便益）。
- **QTranslator を狭めない**: キー付き定数テーブルは将来 `QT_TRANSLATE_NOOP`/gettext へ機械変換可能な形。今回は翻訳フレームワークを導入しない（YAGNI — 対象言語は日本語1つ）。

### 2.2 Qt 標準ボタンの日本語化 — qtbase QTranslator（判断点 #10 確定）

`theme/apply.py` の起動適用系列に **qtbase 日本語 QTranslator のロード**を追加する。ただし以下の規律を必須とする:

- **install-once の冪等化＋QApplication 存在ガード**: `installTranslator` は QApplication 生成**前**に呼ぶと恒久無効（実測: install 前生成の translator は app 生成後も効かない）。また `apply_startup_theme` は `build_main_window` から毎回呼ばれテストスイートで 1 プロセス数十回実行されるため、素朴な実装は translator を蓄積し install ごとに全 widget へ LanguageChange を配送する。**module-level singleton＋install 済みフラグで1回だけ install・QApplication 不在時はスキップし次回呼び出しで自己回復**（Fusion の `vs_fusion_applied` property フラグと同型の規律）。Layer A テストで「2 回呼んでも translator が増えない」冪等性を assert。
- **生存保持**: singleton 化が同時に GC 消失（install 後に Python 参照が切れると外れる）も防ぐ。
- **ロード対象**: `QLibraryInfo.path(TranslationsPath)` から `qtbase_ja`。PySide6 wheel の `translations/qtbase_ja.qm` 実在はテストで検証。ロード失敗時は警告ログのみで起動継続（自前文言は translator 非依存）。
- **効果範囲の正確な理解**: 効くのは Qt 実装の UI — QDialogButtonBox/QMessageBox 標準ボタン（**「はい(&Y)」「いいえ(&N)」とニーモニクス内蔵形で供給**される — 画面表示は「はい(Y)」）・QColorDialog・テキスト編集の標準コンテキストメニュー・Data Explorer の QFileSystemModel 列ヘッダ（Name/Size/…→名前/サイズ/…）。**ネイティブ QFileDialog（getOpenFileName 等）は OS 供給文言で translator の適用外**（日本語 Windows では OS が日本語を出す）。translator は install 時に既存 widget へも遡及適用される。
- **明示制御の上書き**: 標準ボタンでも文言を厳密制御したい箇所は `button().setText(strings.X)` で上書き — QTranslator と両立。G-14 の削除確認ダイアログはこの方式でボタンを「閉じる/キャンセル」に確定（本文動詞と一致）。
- **テスト隔離**: 「install 後に QDialogButtonBox 文言が日本語」のガードテストは **fresh-process**（`test_theme_apply.py` の subprocess パターン流用）で行う — 共有 qapp では他テストの build_main_window 経由の残留 translator と区別できない。逆に、translator 導入後は共有 qapp の全テストで Qt 標準文言が日本語になりうることを前提に、標準文言を assert する既存/将来テストの実行順依存を §6 の追随棚卸しで点検する。
- **realgui の install 経路**: realgui の多く（14/16 本）は `MainWindow(AppViewModel())` 直構築で `apply_startup_theme` を通らない。①ゲートの「QTranslator の効き」実機確認は **build_main_window（実アプリ組立て）経由のテスト/手順で行う**と明示し、直構築テストでの標準ボタン文言は「英語のままでありうる（実行順依存）」ことを既知事項とする。

### 2.3 core 診断・例外文言 — core 内リテラル日本語（UX-08）

core は gui へ依存できない（レイヤー逆転禁止）ため、`strings.py` へは載せず **core 内のリテラルを日本語化**する:

- 対象は付録 core クラスタ（mdf_loader / csv_loader / csv_format_detector / session の診断・例外）＋補完クラスタ core-extra（**format_def.py の検証 ValueError 5 件** — csv_format_dialog のエラーラベルへ生表示・**csv_exporter.py の検証メッセージ 3 件** — export_csv_dialog へ生表示）。
- **format_def_manager.py はスコープ外**: GUI からの呼出サイトが無く（grep 0 件）ユーザー露出経路が存在しない — 防御的例外は原文のまま（露出経路ができた時点で日本語化）。
- 診断プレフィックスは `信号 '{name}':` へ 6 箇所統一（G-03）。`signal_name` フィールドは別途構造化保持済みのため表示文言のみの変更で安全。
- 2 ローダーで同一原文の文言（`File not found or not accessible: {path}`）は同一訳を強制 — **同一性テスト**（両モジュールの該当文言が一致することを assert）で writer のコピー先揺れを恒久防止。
- **文言依存テストの同時更新**は §6 の三本立てプロトコル（負アサート含む）に従う。

### 2.4 ニーモニクス体系（UX-41）— 適用面の確定

**付与する面（全数列挙）**:
1. **メニューバーのトップレベル**: ファイル(&F)・表示(&V)・解析(&A)・ヘルプ(&H)。
2. **メニューバー配下のサブメニュー項目**: 割当は **§3 G-46 の表が全数かつ唯一の出典**（例示ではない）。共有 QAction・動的合成項目は対象外（次項）。2面共有文言は §2.1 の `mn()` 規則でメニューバー側のみ付与。
3. **ダイアログの主要ボタン**（QDialogButtonBox 外のカスタムボタン）: 任意（衝突回避優先）。

**付与しない面（確定）**:
- **コンテキストメニュー9面と共有 QAction（AnalysisActions）**: Windows でもコンテキストメニューのアクセスキー慣行は弱く、G-28 の3面共有定数（メニュー/線右クリック/readout）がニーモニクス込みで同一である制約と、realgui 掴み点の破壊面最小化から**付与しない**。
- **ドックトグル（toggleViewAction 由来）**: text は dock windowTitle の写しで、windowTitle に & を入れると CollapsibleDockTitleBar/レールタブの QLabel（buddy なし）に「(&F)」が生表示され、setText 上書きは windowTitle 変更で Qt が再同期し消える（実測）。**ニーモニクス対象外**とし、§9 の到達性は Alt+V→矢印キーで充足と定義する。
- **テーマ radio 等の動的合成項目**: 同上（矢印キー到達）。

**重複検出テスト（Layer B・実メニュー walk）**: strings.py 内タプルの自己申告方式は実メニューとの乖離（載せ忘れ）に盲目のため採らない。**build_main_window で実 QMenuBar を構築し `menuBar().actions()`→各 QMenu.actions() を walk（サブメニューは再帰）、メニュー単位で `&` 直後文字（小文字化・`&&` 除外）の一意性を assert**する。加えて **G-46 の表と実メニューの付与項目集合が一致**することを assert（付与漏れ・載せ忘れの双方向検出）。コンテキストメニュービルダーは**実在全数** — graph 系の build_context_menu／build_curve_menu／build_axis_menu／build_x_axis_menu／build_cursor_menu／build_readout_menu（build_column_menu 含む再帰）・channel_browser_view／file_browser_view／data_explorer_view の3コンテキストメニュー構築経路 — を注入あり/なし両分岐で walk し「& を含まない」（付与しない規約そのもの）を assert。メニューへの項目追加が自動的に検査対象へ入る構造で、ドリフト不能（ビルダー新設時は walk テストへの登録を実装プランのチェック項目とする）。

### 2.5 変更しないもの（互換性の錨）

- **objectName・QSettings キー**: dock/widget の objectName は `saveState()`/`dockCollapsed` 永続化の識別子 — 一切変更しない（windowTitle 等の表示文言のみ変更）。
- **VM キー・データ値**: `visible_stat_cols` の VM キー（mean/min/max/std/count）・csv_format_dialog の currentText データ値兼用（sec/msec）は据え置き（判断点 #4/R-11）。表示と内部キーの分離導入は据え置き理由ごと対訳表に「意図的英語」として記録。
- **数値フォーマット**: 時刻表示は**固定小数3桁 `{t:.3f} s`（計測 IA で `.4g` から変更済み — 本増分では不変）**・readout 値列 precision（4/6/8）・カーソル即値 `A {t:.3f} s` は現行仕様のまま。オフセット表示のみ R-06 で 4 画面 `{:+.3f} s` へ統一（意図的変更 — §6 に記載）。signal_preview の時間範囲 `.4g` は据え置き（表示のみ R-05 のダッシュ是正）。

### 2.6 揺れ再生産防止の enforcement（設計判断の記録）

theme 増分1 の AST ガードと同型の「strings.py 迂回リテラル検出ガード」は**置かない**。文言は色と違い、`setText` 等へ渡る文字列に正当な動的合成・データ値が多く、heuristic 判定の false-positive 保守コストが利益を上回ると判断。代替の担保: (1) §2.4 の実メニュー walk テスト（メニュー系の追加は自動検査）、(2) strings.py への集約自体（レビューで直書きが目立つ構造）、(3) 増分ごとの spec レビューでの文言点検（本プロジェクトの確立済み運用）。増分E 以降でこの判断を見直す場合は docs/design.md の決定履歴に記録する。

## 4. 表記規約（docs/design.md へ「表記規約」節として転記 — UX-55 の恒久解）

| # | 規約 |
|---|---|
| R-01 | **一次言語は日本語**。原文維持: 固有名詞（ValiSync）・頭字語/規格名（CSV/MDF/XCP/OK）・単位（s/Hz）・記号（Δ/σ/±/A/B）・キー名（Ctrl/Shift/←→）・識別子/ファイル名/dtype/例外原文（{exc}）。 |
| R-02 | **括弧**: 括る内容が ASCII のみ（ショートカット・記号・数値）なら半角括弧＋直前半角スペース（「カンマ (,)」「新規タブ (Ctrl+T)」「上限 ({n})」）。日本語または非 ASCII 記号（Δ 等）を含む補足は全角括弧（「カーソル B（Δ）」「（履歴なし）」「オート（OS に合わせる）」）。全角括弧の RUF001/RUF002 は noqa 付与を既定とし、lint 回避のために表記を歪めない。 |
| R-03 | **三点リーダ**: 後続入力・ダイアログを伴うアクションのみ末尾「…」（U+2026 1文字）。即時実行には付けない（G-39 の実挙動不一致「…」は除去）。ニーモニクスは三点リーダの前（「開く(&O)…」）。 |
| R-04 | **ショートカット表記**: QAction は setShortcut 登録で Qt 自動併記に任せ、ラベルへの手書き併記は禁止（welcome_view.py の「(Ctrl+O)」stale 罠はボタン文言を action から動的合成して根治 — E-3）。ボタン/ツールチップへ併記する場合は「ラベル (Ctrl+O)」— 半角括弧・区切りスペース1つ・キーは + 前後スペースなし。 |
| R-05 | **ダッシュ**: 区切りは em ダッシュ「 — 」（前後半角スペース。「{name} — ValiSync」「信号プレビュー — {key}」）。数値範囲は en ダッシュ「–」スペースなし（0–255・min–max・「{t_min:.3f}–{t_max:.3f} s」— file_browser_vm/signal_preview_vm の範囲表示もこの形へ是正）。 |
| R-06 | **数値・単位**: 数値と単位の間に半角スペース（「+0.500 s」「1024 列」「5 件」）。**オフセット表示は単一テンプレート `{:+.3f} s` を 4 画面（ドラッグ追従 tooltip 1884・適用確認 1962・入力ダイアログ 2385・情報行 2350）で共用** — 1884/1962 の `.3g`→`.3f` は意図的表示変更（§6 に記載・UXG-04 の完全解消）。 |
| R-07 | **助数詞**（UX-11）: 診断・ヒット件数=「件」／信号総数=「N 信号」・フィルタ結果=「{n} 信号中 {m} 件を表示」の単一パターン（当該現物は G-42）／展開系列・チャンネル=「本」／列=「列」／サンプル=「点」／非単調=「箇所」。「ch」はファイル情報等の技術メタ情報に限定。数詞は半角数字＋前後半角スペース（「1信号」直結は禁止）。 |
| R-08 | **動詞形**: メニュー/ボタン=辞書形「〜を開く」（文脈自明の単独動詞「削除」「非表示」許容）／完了=「〜しました」／進行=「〜中…」／失敗=操作の不成立は「〜できませんでした」・内部要因＋詳細併記は「〜に失敗しました: {exc}」／依頼=「〜してください」／確認=「〜しますか？」。**明示 except: ステータスバーの短文表示は名詞止め「〜失敗: {対象}」を許容**（G-13/G-17 — 幅制約の実利）。ボタン対は動詞で対称に（G-19）。 |
| R-09 | **入口動詞と処理動詞の分離**: 「開く」=ユーザー起点の入口ラベル／「読み込む」=処理の記述（進行・完了・失敗）。機能名は「エクスポート」で統一。 |
| R-10 | **記号・句読点**: 疑問符は全角「？」。コロンは半角「: 」＋後スペース。ダイアログ本文は「。」終止・ラベル/メニュー/ツールチップは句点なし。ステータスの列挙連結は「 ・ 」。二重スペースは即修正。診断 # 表示は 1 始まり（E-2）。 |

## 6. テスト戦略（/gui-test-plan 分析）

**変更種別**: 文言（表示文字列）全域置換＋起動時 QTranslator 適用＋意図的表示変更 E-1〜E-5。イベント経路・レイアウト構造の変更なし（文言起因のコントロール寸法変化は生じる — 凍結検証の期待に含める）。

### Layer A（必須・CI）
- `strings.py` テンプレート整合（全テンプレート定数が期待プレースホルダ集合で format 可能）。
- QTranslator: ロード成功（qtbase_ja.qm 実在）・**冪等性（2 回適用で translator が増えない）**。「QDialogButtonBox 文言が日本語」のガードは **fresh-process**（§2.2）。
- core 2 ローダーの同一文言の同一性テスト（G-34）。
- 診断 # 1 始まり（E-2）の表示値テスト。

### Layer B
- **ニーモニクス重複検出＝実メニュー walk**（§2.4 — build_main_window の実 QMenuBar＋コンテキストメニュービルダー両分岐。従来型のタプル自己申告は採らない）。
- E-3 の CTA 動的合成（QAction shortcut 変更がボタン文言へ反映）。

### 既存テストの文言追随 — 三本立てプロトコル（機械一括置換は禁止）
1. **旧文言の全文 grep**（en/mixed の対応表全行）: ヒットサイトを列挙し、**サイト単位で文脈確認して**追随（「All」「テーマ」等の短文言は部分文字列衝突が多く、一括置換はテスト破壊の別経路になる）。
2. **英語トークンの断片 grep**: 旧文言を構成する語（skipped / non-numeric / columns / not accessible 等）で tests/ 全域を grep。**特に負アサート**（`not any("skipped" in d.message ...)` 型 — tests/test_loaders.py:476-479, 591-593・tests/test_demo_mf4.py:227-232）は文言変更で**恒久 vacuous 化しても落ちない** silent false-green。新文言の断片へ書き換えたうえで **sabotage（旧挙動を一時再現）で RED を確認**してから green 化するか、構造化フィールド（signal_name・level）ベースへ移行する。
3. **ja→ja(&X) と非文言変更の明示列挙**: ニーモニクス付与は既に日本語の文言も変える（「開く…」→「開く(&O)…」）— 対応表に **ja→ja 行を必ず含める**。文言に還元できない変更（E-2 の seq 表示）はテストファイル直接指名（§5 E-2）。

**既知の特殊サイト（機械置換対象外・手動書換）**:
- **& 除去・部分文字列比較 4 サイト**: tests/realgui/test_shell_chrome_flow.py:142・test_theme_menu_realclick.py:62・test_analyze_menu_realclick.py:117・tests/gui/test_main_window_menus.py:13,20-21。新形式「ファイル(&F)」では `replace("&","")` イディオムが崩れる（「ファイル(F)」になる）— `re.sub(r"\(&.\)", "", text)` の共有 strip ヘルパへ書き換える。
- **リスト全体等値 3 サイト**: test_data_explorer_realclick.py:88・test_file_browser_realclick.py:99 系・test_channel_browser_realclick.py:105 — 訳語・項目順込みで新文言を直書き。
- 破壊面の規模感（レビュー実測）: realgui 確定 13 サイト/9 ファイル＋Layer B 10+ ファイル（test_context_menus.py 14・「ズームアウト（引き）」6 ほか）。レビューのサイト一覧を実装プランへ初期リストとして引き渡す。

### Layer C（realgui・merge 前 ①ゲート）
- 主要ジャーニー（開く→メニュー操作→右クリック→エクスポート→診断→**Data Explorer**〔QFileSystemModel ヘッダの translator 効き確認〕）を実 OS 入力で通し、日本語化後のメニュー・ダイアログ・ステータスバーのスクショ目視。
- **QTranslator の効き確認は build_main_window 経由のテスト/手順で行う**（§2.2 — 直構築 realgui では英語のままでありうる）。
- Alt→ニーモニクス到達の実機確認（G-46 の全付与項目 — トップレベル4つ＋各メニュー配下の付与項目。§9 と同範囲）。

### 凍結検証（撮影）
- 文言全域置換のため全カタログ状態が変わる。**目視対象は 9 状態 × dark/light = 18 枚の --diff-out 赤マスク**。
- 期待の正確な言明: 「差分は**テキストおよびテキスト起因のコントロール寸法変化**に限られ、**プロット viewport 内はピクセル一致**」。プロット不変は目視でなく機械化 — 各状態でプロット viewport 矩形を geometry から取得し crop 領域比較する比較モードを実装タスクに含める。
- 手順: 変更前後比較（18 枚目視＋viewport crop 機械一致）→ 新ベースラインへ一括昇格 → 再撮影 compare exit 0（決定性）→ DesignSync 再同期。想定差分リストは §5 の E-1〜E-5。

## 8. 実装増分の分割方針（writing-plans への入力）

**PR 構成の確定: D-1 全体を単一 feature ブランチ（内部増分＝コミット列）とし、①ゲート・撮影比較・ベースライン昇格・DesignSync 再同期はブランチ完了時の 1 回**。docs/design.md の反復単位（1 反復 = 1 feature ブランチ）と一致し、§6 の「変更前」基準が漂流しない。中間コミットは「エリア単位で文言＋テスト追随が閉じて品質ゲート緑」を保つ（日英混在の中間状態は許容 — 表示上の過渡であってテスト赤ではない）。

1. **基盤**: `strings.py` 新設＋テンプレート/冪等ガードテスト＋QTranslator 適用（§2.2 の規律込み）。
2. **シェル**: メニューバー・ツールバー・ステータス・ドック名（3面同期）・Welcome（E-3）・About・**workers（G-41 — BusyOverlay 文言）**。
3. **ダイアログ**: export_csv（G-19/E-1）・csv_format・expansion・signal_preview・確認ダイアログ群（G-14 ボタン setText）。
4. **グラフ系**: コンテキストメニュー9面・analysis_actions・cursor_readout・offscale_badge・タブ既定名（G-40）・オフセット書式（E-4）。
5. **ブラウザ・診断**: channel/file/data_explorer（G-39 含む）・diagnostics（フィルタ・列ヘッダ・E-2）・**VM 文言（G-42・ツールチップ群・signal_preview_vm）・adapters（G-43 列ヘッダ）**。
6. **core 診断**: loaders/session/**format_def（G-44）/csv_exporter（G-45）** の日本語化＋同一性テスト＋負アサート棚卸し（§6-2）。
7. **凍結・同期**: 実メニュー walk テスト最終確認→撮影前後比較（18 枚＋viewport crop）→ベースライン昇格→DesignSync 再同期→docs/design.md（表記規約節＋決定履歴）。

各増分内で「文言＋既存テスト追随＋realgui 追随」を不可分に扱う（§7 リスク対策）。**対応表は §3＋§4 から機械再生成し、付録の提案列は出典にしない**（§3 冒頭規則）。
