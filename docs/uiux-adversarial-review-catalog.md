### UX-01 ✅解消（PR #135・軸アイデンティティ契約 Stage A） — Y軸ラベルの単位が「最後に追加した信号の単位」で上書きされ、名前と単位が別信号の捏造ペアになる（VehSpdInternal (Nm) / YawRate (Nm)）

- **検出レンズ**: visual, ia, interaction, onboarding, a11y, domain, micro
- **増分**: 新規（最小修正=unit first-wins 化＋全経路再計算は独立バグ修正。単位混在の重ねを許すか・混在時ラベル方針のみ増分E 比較データモデル設計と併せて確定）

7レンズ全てが独立に検出した最重問題。同一軸に単位の異なる信号を重ねる（既定の追加経路がまさにそうなる）と、軸ラベルが「最初の信号の名前＋最後に追加した信号の単位」という物理的に誤った合成になる。ADAS エンジニアは軸目盛の単位をラベルから読むため、km/h の速度を Nm と誤読する経路が直接存在し、スクショを報告書に貼れば誤記が伝播する。読み値ペインの正しい単位と同一画面で食い違い、どちらを信じるべきか判断できない。name/unit の採用規則が非対称（first-wins vs last-wins）なのが根因で、意図しない実装である可能性が高い

**修正方向**: 最小修正: unit も first-wins に揃え、代表エントリの name/unit を同一信号から取る。根本解決: 軸上の全信号の単位が一致する場合のみ単位表示し、混在時は単位を落とす（または「(混在)」明示）。単位再計算は追加・削除・移動・overwrite の全経路で行う。単位混在の重ね自体を許すかは増分E の比較データモデル設計と併せて確定

**検証所見**: スクショ実見で「VehSpdInternal (Nm)」「YawRate (Nm)」の文字列と、同一画面の読み値ペイン [km/h]/[Nm]・Channel Browser Unit 列 (km/h, deg/s, Nm) の矛盾並置を確認。コードも指摘どおり: add_signal_to_axis で unit は無条件上書き (last-wins)・name は first-wins の非対称 (コメントは name の first-wins のみ文書化＝unit 側は無自覚の疑い濃厚。overwrite_axis は name/unit を対でクリアしており作者意図は「代表信号のペア」)。move_entry_to_new_axis は新軸に name/unit 未設定で無ラベル軸を生む。さらに主要追加導線 _add_to_active_panel→add_signal は常に軸0へ join するため既定操作の2本目で即発症＝スクショは作為でなく代表的再現。R15 設計が凡例撤去時に「軸ラベルが識別を補完」と明記した指定識別面が物理的誤単位を表示する点で実害は重い。既存計画 (D/E/F・audit catalog) に該当なし。

**証拠**: src/valisync/gui/viewmodels/graph_panel_vm.py:261-271 (unit last-wins vs name first-wins)・:285-297 (overwrite_axis は対でクリア=意図の傍証)・:718-740 (move_entry_to_new_axis 新軸 name/unit 未設定)、src/valisync/gui/viewmodels/y_axis_vm.py:19-29 (既定 ""), src/valisync/gui/views/graph_panel_view.py:1195-1196 (setLabel 描画), src/valisync/gui/views/main_window.py:490-498 (既定追加経路→axis 0 join)。スクショ: crops/r04_axislabel.png・crops/r05_axislabel.png・shots_real/r04_cursor_readout.png (読み値 [km/h] と軸 (Nm) の同画面矛盾)

### UX-02 ✅解消（PR #135・軸アイデンティティ契約 Stage A） — 「新しい軸へ移動」した曲線が 0–1 デフォルトレンジ・無ラベルの新軸に落ちて画面から消える

- **検出レンズ**: visual, ia, interaction, onboarding, domain
- **増分**: 新規

曲線右クリック→「新しい軸へ移動」（PR #63 の正規導線）を実行すると、値域が 0..1 に収まらない信号は移動した瞬間に視界から消え、新軸は名前も単位も無いため何の軸かも判別不能。ユーザーには「移動したら曲線が消えた」壊れ方・データ消失に見え、軸右クリック→オートフィットを知らなければ復帰できない。多軸分配（速度/トルク/ヨーレートの単位別分離）は ADAS 解析の基本操作であり主要フローの阻害。5レンズが独立に検出し、うち2レンズは critical 評価

**修正方向**: move_entry_to_new_axis で (a) 移動エントリに対する auto-fit（create_new_axis→add_signal_to_axis と同じ経路）、(b) name/unit の伝搬、(c) 移動元軸の name/unit 再計算を行い、作成系と挙動を揃える

**検証所見**: 実見＋コードで全面裏付け。r03/r04 実機スクショで中段・下段リージョンが無ラベル・目盛 0.2–0.8（pyqtgraph ViewBox 既定 0..1）のまま曲線不可視、一方 r04 の読み値ペインは同信号の値（SetSpeed 80–80・TargetAccel −2.58..0・YawRate ±3.16–3.23）を表示しデータ実在を証明。コード: move_entry_to_new_axis は YAxisVM を y_range=None/name=""/unit="" で生成し _auto_fit_ranges 非呼出・name/unit 非伝搬（add_signal_to_axis は両方行う非対称）。view は y_range None の軸に setYRange せず、全 ViewBox が disableAutoRange() のため 0..1 既定に留まり値域外の曲線は不可視。メニュー配線 (view:2169) で実ユーザー経路がこの VM メソッドに直結。by-design 反証も棄却: spec DP17 は「D&D 新軸と同一 VM 経路」と等価性を主張しており（D&D 経路はフィット＋ラベルする）実装が spec 自身と矛盾、テストも再割当/compact/cache-bust のみで no-fit を lock していない＝見落としバグ。緩和策（軸右クリック→オートフィット・後続 add_signal で偶発的に全 None 軸がフィット）は存在するが UI が示唆せず、多軸分配は ADAS 解析の基本操作の正規導線で 100% 再現。

**証拠**: src/valisync/gui/viewmodels/graph_panel_vm.py:718-740 (move_entry_to_new_axis: y_range=None の新軸・_auto_fit_ranges 非呼出・name/unit 非伝搬) vs 246-275 (add_signal_to_axis: unit/name 伝搬 261-271＋_auto_fit_ranges 273)。src/valisync/gui/views/graph_panel_view.py:973-978 (y_range None は setYRange スキップ)・1159 (vb.disableAutoRange())・2169-2170 (メニュー「新しい軸へ移動」→vm.move_entry_to_new_axis 配線)。src/valisync/gui/viewmodels/y_axis_vm.py:13-29 (既定 y_range=None/name=""/unit="")。docs/superpowers/specs/2026-07-09-gui-plot-analysis-controls-design.md:74 (DP17「D&D 新軸と同一 VM 経路」)。tests/gui/test_graph_panel_multi_axis.py:1349-1390 (no-fit を lock するテスト無し)。実見: shots_real/r03_multiaxis.png・r04_cursor_readout.png (0.2–0.8 無ラベル軸×2＋readout に min–max 実値)。

### UX-03 ✅解消（PR #135・軸アイデンティティ契約 Stage A） — 同一Y軸に後から追加した信号がオートフィット対象外で完全に不可視になる（可視チェックONのまま描画されない）

- **検出レンズ**: ia, interaction, onboarding, micro
- **増分**: 新規

最初の信号で軸が確定した後、同じ軸へ追加した信号はレンジ再計算されず、スケールが違うと影も形もなく消える。readout・軸メニューには存在するため「信号がフラット/死んでいる/データが無い」と誤読し、イベント（AEB 作動等）を見落とす。off-scale の手掛かり（インジケータ・警告）も皆無。km/h・deg/s・Nm 等スケール混在が常態の ADAS 計測で、187ch から信号群を素早く並べる中核ジャーニーの一歩目が壊れる。D&D 既定（新軸作成）では起きずメニュー/ボタン経路でだけ起きる不一致も混乱を増やす

**修正方向**: RN-02 の Y 版を実装: 軸ごとに auto フラグを持ち、手動レンジ設定までは追加/join/overwrite で全載籍信号の和集合へ再フィット（X との対称化）。手動設定後の追加はオフスケール矢印バッジ等で「レンジ外に曲線あり」を通知するか一度だけオートフィット提案。スケール乖離が大きい場合の新軸自動分配は増分E の設計と整合させて検討

**検証所見**: スクショ実見＋コード裏取りの両方で完全に事実。03_cursor.png で csv_1::VehSpd が readout に載る (A=72・min–max 0–118.8) のにプロットに1ピクセルも描かれず Y軸は EngineSpeed 800–2275 のまま。c02_ctx_axis.png で軸0の4信号全チェックONだが r03 で可視は2本のみ。根因は graph_panel_vm.py:1332-1351 の `if axis.y_range is None:` ゲート — _fit_axis (664-671) が初回 add で具体値を格納するため以後恒久的に再フィット不能。RN-02 の auto フラグ (_x_range_is_auto) は X のみで Y 版が存在しない。add_signal docstring (241-242) の「y_range も和集合にフィット」は実装と乖離。実ユーザー経路も確認: Channel Browser 右クリック「Add to Active Panel」(main_window.py:498 → add_signal → 軸0)・Ctrl+ドロップ join (graph_panel_view.py:2065-2070)。加えて plain ドロップの overwrite_axis (285-297) も y_range を残すため唯一の新信号すら不可視になりうる。プロット面ドロップのみ create_new_axis で無事 → 経路間不一致も正。by-design 検証: RN-02 spec (2026-07-05-rendering-correctness-design.md:75,99) は「y_range は対象外・既存挙動を維持」というスコープ先送りであり UX 決定ではない。増分E (比較データモデル) は Y 再フィット/オフスケール通知を含まず、むしろ E の同名信号自動重ねが join 経路を主動線化するため本件は E の前提修正。緩和策 (軸メニューのオートフィット・readout min–max) は疑ってから使う回復手段で、無言不可視への手掛かりゼロという核心は変わらない。

**証拠**: graph_panel_vm.py:1332-1351 (`if axis.y_range is None:` ゲート・Y 再フィット初回のみ)／同 664-671 `_fit_axis` が具体値格納でゲート恒久閉鎖／同 238-244 docstring 乖離／同 285-297 overwrite_axis が y_range 非クリア／graph_panel_view.py:2065-2080 (Ctrl join=add_signal_to_axis・plain=overwrite_axis・プロット面のみ create_new_axis)／main_window.py:307-309,496-498 (「Add to Active Panel」→add_signal→軸0)／specs/2026-07-05-rendering-correctness-design.md:75,99 (Y はスコープ除外の明記)／実見: shots_catalog/03_cursor.png (VehSpd readout 記載・曲線0px・軸 800–2275)・crops/02_bottom.png (下端にも無し)・shots_real/c02_ctx_axis.png (4信号チェックON) vs r03_multiaxis.png (可視2本・軸0≈57–80)

## デザイン案（3 方向 17 案 → 統合判定で推奨 6 案）

polish（制約内洗練）/ workflow（構造変更）/ radical（抜本再考）の3方向で独立生成した 17 案を、
横断判定（課題カバレッジ・相互整合・増分 D/E/F との整合・段階実施性）で 6 案の推奨パッケージへ統合した。

**統合判断**: 17案は実質「polish/workflow 案 ⊂ radical 案」の包含構造にあり、radical 6案がいずれも段階移行パス込みで下位案の上位互換だったため、radical 側6案を推奨パッケージとし重複11案を吸収先明示で見送った(安易な全面リライトではなく、各案の第1段が小粒バグ修正/トークン変更で単独 merge 可能なことを採用条件とした)。真の相互排他は1組のみ: 軸まわりで「軸アイデンティティ契約」(既定挙動不変・表示の真実性のみ規定)vs「単位が軸を決める」(既定 add を自動ルーティング化)。前者を採用 — critical 3件の根治は共通だが、後者の挙動変更は UX-01 の検証結果自身が「混在可否は増分E と併せて確定」とした判断領域であり、E 前に固定すべきでない。依存関係は (1) 軸契約 Stage A → 増分E(同名重ねの前提)、(2) カーソルタブ所有化 → F の A–B 範囲エクスポート/スナップショット共有、(3) 増分0 の Okabe-Ito 色相確定 → E-2 の色相ファミリー(二度変えを防ぐため増分0 で最終値として確定)、(4) 文言OS(D-1) → 14(b)(c)/E/F の全新設 UI 文言 — の4本で、推奨の実施順(増分0＋Stage A → 14(a) → D → 14(b)(c)/Stage B-C/E-0 並行 → E → F)はこれを全て満たす。一貫性の担保として、記録済み決定の supersede が必要な3点(spec-B 案b のプレースホルダ専用決定→凡例モード、FB-02 チップ表示→件数内蔵フィルタ、絵文字置換 defer の実行)は各増分着手時にユーザー確認を必須とし、凍結スクショベースラインの無効化はどの案でも不可避のため増分0 で最初に一度払い、以後は増分単位の更新として扱う。confirmed の critical 3件・major 全件はこの6案で漏れなくカバーされることを突合済み(minor/polish も UX-24 のゾーンヒント任意化を除き全て回収)。

### 推奨1: 「知覚の床」バンドル — トークン・寸法・QSS の即効一斉是正

- **方向/スコープ**: radical / token
- **解く課題**: 知覚基盤の一斉是正（増分0） — 解く課題: UX-12(エラー文字 3.0:1)・UX-18(tab10 CVD 非対応・暫定)・UX-21(初期ドック配分・部分)・UX-27(読み値ヘッダ 4.4:1/9px)・UX-29(Name 列省略)・UX-35(アクセント値衝突)・UX-38(24px 未満ターゲット)・UX-39(グリッド不可視)・UX-42(フロッピー誤メタファ)・UX-49(未フォーカス枠不可視)
- **採用理由**: 案1(読める計測面)＋案2(カラーセマンティクス v2)の全内容に、グッドデフォルト/Lucide 案のクイックウィン(初期 resizeDocks・列幅既定・24px 当たり判定・export アイコン差替)まで含む上位互換。全変更が実測数値根拠付き(error 3.02:1→7.08:1・text_secondary 4.44→5.81:1・grid_alpha は 90–120 では 3:1 に届かない実測に基づく 150)で、既存の凍結比較スクリプト＋値分岐テスト流儀にそのまま乗る最小リスク構成。UX-12/18/27/29/35/38/39/42/49 と UX-21 の応急部分を一括解消し、凍結スクショベースライン無効化のコストを最初に一度だけ払える。
- **実施順序**: 増分0 として最優先(軸バグ修正と並行可・1〜2 PR)。パレットは「暫定」でなく Okabe-Ito 色相を最終確定として入れ、増分E の「ファイル=色相ファミリー」は同一色相基準で設計する制約を docs/design.md に同時記録 — これで案2 が警告した「E で二度変わる」リスクを消す。既存 D/E/F のどれにも依存しない独立増分。

構造変更に着手する前に、tokens.py/qss.py/初期寸法の1点変更で解ける知覚系10課題を「増分0」として1〜2 PR で一括是正する。全て実測済みの数値根拠があり、既存の凍結比較スクリプト＋値分岐テスト流儀にそのまま乗るため、後続の抜本案 (2〜6) の土台を汚さずに体感品質を先に引き上げる。

**具体化**:

**トレードオフ**: 凍結スクショベースラインが一括で無効化されるため撮り直し1回分のコストが発生する (どの案を選んでも一度は払うので最初に払うのが最安)。パレット暫定差替は増分E の本設計 (ファイル=色相) で再度変わりうる二度手間リスクがあるが、CVD 制約の実機検証を先行できる利点が勝る。small_px 10 化は読み値ペインの情報密度をわずかに下げる。当たり判定拡張は隣接コントロールとの重なり検証 (Layer C) が必要。

### 推奨2: 軸アイデンティティ契約 — 単位整合・Y和集合フィット・軸ヘッダチップ

- **方向/スコープ**: radical / structure
- **解く課題**: 多軸表示の信頼性崩壊 — 解く課題: UX-01(軸単位の捏造 critical)・UX-02(新軸移動で曲線消失 critical)・UX-03(追加信号の無言不可視 critical)・UX-22(曲線↔名前の対応手段なし)・UX-23(偽 0-1 目盛と誤誘導ヒント・空状態部分)・UX-47(リージョン区切り/ラベル生切り)・UX-54(単位表記揺れ)
- **採用理由**: critical 3件(UX-01 軸単位の捏造・UX-02 新軸移動で曲線消失・UX-03 追加信号の無言不可視)を単一の「軸表示は常に真実」契約で根治する唯一の案。競合する案8(単位ベース軸ルーティング)と違い既定 add の挙動(軸0 join)を変えないため既存習熟・テスト資産への波及が最小で、「単位混在の重ね可否」という設計判断を増分E に正しく残す。Stage B/C で UX-22/23/47/54 も回収。増分E の同名信号自動重ね(join 主動線化)は Stage A の Y和集合フィット無しでは UX-03 を踏むため、E の明示的な前提修正。
- **実施順序**: Stage A(name/unit 全経路再計算・Y auto フラグ和集合フィット・move=create 同一経路化・オフスケールバッジ)は独立バグ修正サブスペックとして即着手・最初に merge。Stage B(軸ヘッダチップ・リージョン区切り・[unit] 統一)/Stage C(空状態)は独立増分「axis-identity」として増分D と並行可。E-2(比較モデル)より前に Stage A 完結が必須。

「軸に表示される単位・範囲・所属は常に真実である」という契約を VM に導入して critical 3件を根治し、その可視化として回転軸ラベルを『色ドット列＋合意単位』の軸ヘッダチップへ置換する。空軸・空パネルの偽目盛は廃止。増分E (同名信号の同軸重ねが主動線になる) の前提修正として E より先に完結させる。

**具体化**:

### 推奨4: 増分D 再定義 — 文言オペレーティングシステム＋通知センター型診断＋ドック三態

- **方向/スコープ**: radical / structure
- **解く課題**: クローム/診断の言語分裂と状態表示の嘘（増分D の再定義） — 解く課題: UX-06(フィルタ状態非表示/Clear 同形)・UX-07(メッセージ列読めない)・UX-08(診断文言の日英混在)・UX-10(UI 全域の言語割れ)・UX-11(信号/チャンネル用語揺れ)・UX-20(同一操作の別文言)・UX-31(info カウント矛盾)・UX-32(Sync X/読み値の様式不統一)・UX-34(アイコン3系統/⚠不可視/赤✕常駐)・UX-36(専門用語未説明)・UX-41(ニーモニクス欠落)・UX-44(シェブロン方向固定)・UX-45(レール時トグル矛盾)・UX-50/51/55(表記揺れ)・UX-21 の診断既定部分

**トレードオフ**: 文言全数入替でスクショ比較ベースの凍結検証がほぼ全滅 — 撮影ベースライン再作成を D で一括して払う (案1 と合わせて2回になるのを許容するか、案1 を D 直前に寄せて1回に圧縮するかは計画時判断)。診断詳細ペインは高さを食うため既定折りたたみとセットが前提。三態トグルは Qt の toggleViewAction 内部状態管理と衝突するためカスタム QAction 化の実装コストがある。文言テーブルは初期実装がやや重いが、増分E/F で新設される全 UI がテーブル経由になり揺れの再生産を構造的に防ぐ。

### 推奨5: 単一データサイドバー＋ファイル=色相ファミリーの比較モデル

- **方向/スコープ**: radical / radical
- **解く課題**: データ IA の三分裂と比較データモデル（増分E の具体化） — 解く課題: UX-19(mf4_1:: 露出)・UX-21(File Browser 面積・恒久解消)・UX-29(Data Explorer 列幅側)・UX-30(別窓の役割不明/重複)・UX-11(ドック名の用語統一)・UX-18(ファイル=色の CVD 制約注入)＋E 既定スコープ (基準指定・同名信号の自動重ね)

**具体化**:

### 推奨6: 入口と出口の再設計 — 安全な取込・範囲付き出力・名前付きセッション再開ハブ

- **方向/スコープ**: radical / radical
- **解く課題**: ジャーニー両端 — 取込の安全化と再開・共有ハブ（増分F の再定義） — 解く課題: UX-05(CSV 列番号 off-by-one)・UX-23(Welcome/新規タブの空状態・再開部分)・UX-28(エクスポート範囲指定なし)・UX-43(プレビュー無ラベル)＋F 既定スコープ (スナップショット共有・セッション)
- **採用理由**: 案11(区間エクスポート)を F-0 として完全包含しつつ、増分F 既定スコープ(スナップショット共有・セッション)を .vsession 明示ファイル([[followup_settings_iniformat]] の罠回避と共有可能性を両立)＋Welcome 再開ハブとして具体化する。UX-05 は LD-01 spec が明記しながら実装脱落した仕様の履行であり議論の余地がなく、UX-28 の範囲エクスポートは 14 のカーソルタブ所有化により「どのパネルの A–B か」の曖昧さが構造的に消えた状態で載る。スナップショット共有を 13-A(軸単位の真実性)と 14(Δ 表示検証)の後に置く順序制約を自ら明示しており、誤単位スクショの報告書伝播経路を先に閉じる判断が正しい。
- **実施順序**: 増分F 本体・最後尾。ただし F-0(CSV 列ハイライト＋0始まりヘッダ・範囲ラジオ・選択数フッター・プレビューラベル)は独立ダイアログ改修として D/E と並行先行可 — UX-05 はデータ破損直結のため F-0 だけ切り出して早期に入れる価値がある。F-1(.vsession 永続化・形式バージョニング設計が先行判断)→ F-2(Welcome 再開ハブ＋スナップショット共有)。F-2 は 13 Stage A・14(a)(b) 完了が前提条件。

「開く→見る→書き出す→翌日続きから」のジャーニー両端を再設計する。CSV 取込はスピン連動の列ハイライトで off-by-one を構造的に潰し、エクスポートに範囲指定 (全期間/表示範囲/カーソル A–B) を追加、タブ/軸/カーソル/オフセットを .vsession に永続化して Welcome を『最近のセッション』カードの再開ハブにする。

**具体化**:

取込 (安全化): CSV フォーマットダイアログはプレビュー列ヘッダを 0 始まり「0: t」表記に統一し、時間列/信号列スピンの変更で該当列をリアルタイムに色ハイライト (時間列=cursor_a 系・信号列=drop 系の面色) — 番号の読み替え自体を不要にする。これは LD-01 設計 spec が明記しながら実装で無言脱落した項目の履行。信号プレビュー窓には Time (s)・信号名＋[unit] ラベルを付与 (メタデータ源は VM に既存)。出力 (範囲と確認): エクスポートダイアログに範囲ラジオ [全期間 / 現在の表示範囲 / カーソル A–B]（CsvExportOptions に start/end 追加・カーソル A–B は案3 の計測ステートから取得）、フッターに「N 信号を選択中」常時表示。CANape/MDA の範囲指定標準に並ぶ。セッション (再開): File>セッションを保存/開く — 読込ファイル一覧・タブ/パネル/軸割当・色・カーソル A/B・時間オフセット・折りたたみ状態を明示ファイル .vsession に永続化 (QSettings は使わない — [[followup_settings_iniformat]] の罠回避と共有可能性のため)。ファイル移動/欠損時は差し替えダイアログで再解決。Welcome 画面を [計測ファイルを開く] + [最近のセッション] カード2列に再構成し、HILS 解析の「昨日の続き」を1クリック再開に。新規タブの空状態は案2 Stage C の統一言語に加え「前のタブの時間窓を引き継ぐ」トグルを提示。共有 (スナップショット): 現在ビュー＋読み値表を PNG/クリップボードへ書き出す「スナップショットを共有」— ただし軸単位の真実性 (案2 Stage A) と Δ 表示の検証 (案3) の後に載せる順序制約を明示 (誤単位スクショが報告書へ伝播する UX-01 の経路を先に閉じる)。移行パス: F-0 取込/出力ダイアログ改修 (独立・先行可)→F-1 セッション永続化→F-2 Welcome 再開ハブ＋スナップショット共有。

**トレードオフ**: セッション実装コストの大半はファイル再解決 UI (欠損・移動・別マシン共有) で、永続化層の形式設計 (バージョニング・前方互換) が先行判断として重い。スナップショット共有は案2/案3 への依存を持つため F 内で最後にしか置けない (依存を無視して先行すると誤単位・Δ 非表示の共有物を量産する)。空状態の偽軸除去 (案2 と共通) は 0..1 レンジ前提の既存テストの広い書き換えを伴う。範囲付きエクスポートは統合タイムライン×範囲スライスの相互作用 (マルチレート MDF の loud-fail 既存仕様) の再検証が必要。
