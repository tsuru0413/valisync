# GEMINI.md

このプロジェクトの指示、規約、およびワークフローは **[CLAUDE.md](./CLAUDE.md)** をマスターとして管理されています。
Gemini CLI は、常に `CLAUDE.md` に記載されているガイドライン、開発フローを最優先の指示として遵守してください。詳細なコンテキストやタスク進捗状況（Phase テーブル）、各仕様の詳細については `CLAUDE.md` および `.kiro/specs/` を参照してください。

ただし、**ポインタによる認識漏れを防ぐため、エージェントが必ず守るべきクリティカルな行動規範（AIの振る舞いに関するルール）はここに直接記載し、これを最優先で遵守します。**

## 1. 使用言語 (Language Policy)
- **ドキュメント・コード (英語)**: `.md` ファイル (`requirements.md`, `design.md`, `tasks.md`, `overview.md` など)、ソースコードのコメント、コミットメッセージは、技術的な正確性と一貫性を保つため **必ず英語 (English)** で記述すること。
- **コミュニケーション (日本語)**: ユーザー（人間）とのチャット対話、質問、作業報告は **必ず日本語 (Japanese)** で行うこと。

## 2. ブランチとコミットのルール
- **`main` ブランチへの直接コミットは厳禁**。
- 実装や修正を始める前には、必ず `git checkout -b feature/<spec-name>` または `fix/<bug-name>` で作業ブランチを作成すること。
- 1つのタスク（または小規模な修正）が完了したら、その都度こまめにコミットすること。

## 3. 開発プロセス (TDD と検証)
- **TDD (Test-Driven Development) の徹底**: 新機能やバグ修正を行う際は、**必ず先に失敗するテストを書き (Red)**、テストを実行して失敗を確認してから、それをパスする最小限の実装 (Green) を行うこと。テストを書かずに実装を始めてはならない。
- **完了前の確実な検証 (Verification Before Completion)**: 「テストが通るはず」「直ったはず」といった推測で完了を報告してはならない。必ず実際にテストやビルドコマンドを実行し、その成功出力を確認した上で報告すること。

## 4. 品質ゲート (Quality Gates)
コードを変更してコミットする前には、以下のコマンドを**すべて**実行し、成功（エラーなし）を確認すること。
```bash
uv lock --check
uv run ruff check
uv run ruff format --check
uv run mypy src/
uv run pytest
```
※ CI でも同等のチェックが行われます。ローカルでパスしない限り push や PR 作成は行わないこと。

## 5. 新機能の着手手順 (Overview-first)
新しい仕様 (spec) の策定や大きな変更を始める際は、いきなりコードを書いたり `.kiro/specs/` を変更したりせず、まずユーザーと対話し、合意した要件を `docs/<spec-name>-overview.md` として書き出すこと。これを基に Kiro が詳細な spec を生成する。

## 6. その他の方針
- 推測でバグを直さない。原因が不明な場合は `systematic-debugging` スキルを活用し、エラーログの収集やログ出力による証拠集めを最優先する。
- ユーザーに提案を行う際は、「症状の隠蔽」「症状の緩和」「根本解決」のいずれのアプローチであるかを明示し、常に根本解決を優先する。
